# 智能网络安全分析平台 PRD

## Goal

建设一个面向企业内部安全团队的智能网络安全分析平台。平台以自然语言智能体为主要入口，通过 LLM 完成意图识别、结构化任务规划和结果解释，再由后端受控任务编排器执行固定安全工作流，覆盖钓鱼邮件分析、资产发现、存活探测、端口识别、漏洞扫描、弱口令扫描、防火墙/Web 日志分析、报告导出、审计闭环和安全态势大屏。

本 PRD 来源于 `trellis_security_analysis_platform_task.md`，用于后续 Trellis 拆解和研发排期。当前阶段只定义产品需求，不进入实现。

## Users

### Primary User

* 企业内部安全团队的安全工程师：通过自然语言下发安全任务，查看任务结果、证据、报告和大屏。

### Supporting Users

* 平台管理员：管理用户、角色、资产组白名单、工具配置、LLM Provider、全局策略、日志源、邮件网关和 kill switch。
* 安全审计员：查看任务、审计日志、工具参数、原始证据和报告；不发起扫描或修改配置。
* 管理层只读用户：查看大屏和汇总报表；不发起任务，不查看敏感原始输出。

### Explicitly Not Served

* 普通员工自助扫描。
* 外部客户自助使用。
* 多租户 MSSP 场景。
* 未登录或匿名公网用户。

## Requirements

### 1. Natural Language Agent Entry

* 平台主入口应是自然语言智能体对话框。
* 安全工程师输入自然语言后，系统应完成意图识别、结构化任务计划生成、权限与白名单校验、任务创建、工具执行、结果结构化保存和报告摘要生成。
* LLM 不得直接执行扫描工具或绕过后端校验；后端必须重新校验权限、资产白名单、安全策略和工具参数 schema。
* 当意图不确定或关键字段缺失时，任务应进入 `NEEDS_CLARIFICATION`，由系统提出明确澄清问题。

MVP 结构化任务计划采用步骤级计划：

* MVP 采用自研后端受控任务编排器，并兼容 nanobot 风格的结构化计划和工具协议，不绑定具体 nanobot 实现。
* LLM 只负责输出结构化计划 JSON、澄清问题和结果解释；扫描工具、辅助命令、参数展开、权限判断和任务状态流转均由后端控制。
* LLM 应输出工作流类型和有序步骤列表，例如 `Subfinder -> Httpx -> Nmap -> Nuclei`。
* 每个步骤应包含步骤 ID、步骤类型、输入引用、目标资产组或目标引用、是否需要确认、上游依赖和用户可读说明。
* LLM 可以建议步骤级意图和候选目标，但不得直接决定最终工具参数。
* 每一步具体工具参数必须由后端 schema 校验、补全和强制约束。
* 后端可以删除、调整或阻断不符合权限、白名单、安全策略、速率限制或工具 schema 的步骤。
* 结构化任务计划、后端校验结果、用户确认记录和最终执行计划必须进入审计日志。

### 2. YOLO Mode

* YOLO 模式开启后，在授权资产范围内可跳过人工二次确认直接执行任务。
* YOLO 不得跳过权限校验、白名单校验、工具参数校验、速率限制、超时限制和审计记录。
* 如果 LLM 意图识别不确定，YOLO 不得猜测执行；系统应展示待执行操作或要求用户选择资产组，用户确认后才能继续。
* MVP 中 YOLO 默认关闭。
* 只有拥有 `task:yolo_execute` 权限点的安全工程师才可在单个任务上开启 YOLO。
* 管理员可授予或撤销 `task:yolo_execute` 权限，但不应把 YOLO 作为全局默认开启策略。
* YOLO 开关应按任务记录，不自动继承到后续任务。
* 即使用户具备 YOLO 权限，涉及高风险工具强度、日志脱敏事件导出、LLM 意图不确定、目标范围不明确或白名单校验失败时，系统仍必须要求确认或阻断执行。
* YOLO 开启状态、授权来源、跳过确认的步骤和后端校验结果必须进入审计日志。

### 3. Authorization And Asset Scope

* 平台应使用资产组白名单控制执行范围。
* 白名单类型应支持域名、IP/CIDR 和资产组。
* 白名单应按资产组绑定到用户或角色。
* 每次任务执行前都必须重新做目标归属校验，不能只依赖 LLM 规划阶段校验。
* 域名白名单默认允许执行子域发现。
* 子域发现结果应回写为已发现资产或待确认资产，不得自动扩大到未授权根域。
* 白名单变更必须记录审计日志。

MVP 资产组模型采用发现资产待确认模型：

* 资产组是权限和扫描范围的基本边界。
* 资产组可包含管理员显式配置的域名、IP、CIDR 和已确认资产。
* 子域发现、日志分析、端口识别或其他工作流发现的新资产，应先进入 `DISCOVERED_PENDING_CONFIRMATION` 待确认状态。
* 待确认资产可在资产库和大屏中展示为发现结果，但不得自动进入可扫描范围。
* 子域发现得到的资产在管理员确认前不得作为端口扫描、漏洞扫描或弱口令扫描目标。
* 管理员确认后，资产状态变为 `CONFIRMED`，并可绑定到对应资产组白名单。
* 管理员拒绝后，资产状态变为 `REJECTED`，后续任务不得将其作为授权目标。
* 如果发现资产无法匹配任何已授权根域或 CIDR，应标记为 `OUT_OF_SCOPE_DISCOVERED`，仅作为证据或日志事件展示。
* 日志中出现的新目标 IP / 域名不得自动纳入资产组白名单，也不得触发自动扫描。
* 资产确认、拒绝、资产组绑定和白名单变更必须记录审计日志。

MVP 权限模型采用固定角色 + 细粒度权限点：

* 系统内置四个默认角色模板：安全工程师、平台管理员、安全审计员、管理层只读用户。
* 默认角色用于覆盖常见使用方式；高风险能力必须通过独立权限点显式授予。
* 安全工程师默认可在已授权资产组内创建任务、发起固定安全工作流、查看授权任务结果、查看脱敏报告和大屏数据。
* 安全工程师在授权资产范围内发起弱口令扫描后，可在执行期或短时间窗口内查看和导出该任务的弱口令明文，不需要独立明文查看或导出权限。
* 平台管理员默认可管理用户、角色、资产组白名单、工具配置、日志源、邮件网关、LLM Provider、全局策略和 kill switch。
* 安全审计员默认只读，可查看任务、审计日志、工具参数、原始证据引用和报告；不得发起扫描、修改配置或查看明文弱密码。
* 管理层只读用户默认只能查看大屏和汇总报表；不得发起任务，不得查看敏感原始输出、邮件样本、明文弱密码或工具原始输出。
* 独立权限点至少包括：`task:create`、`task:yolo_execute`、`asset_group:manage`、`tool_config:manage`、`llm_provider:manage`、`log_source:manage`、`raw_evidence:view`、`log_event:export`、`report:export`、`audit_log:view`、`kill_switch:operate`。
* 日志脱敏事件导出、LLM Provider 管理和 kill switch 操作必须是独立权限点，不能仅因拥有管理员或安全工程师角色而默认获得。
* 所有权限拒绝、敏感查看、复制、导出和配置变更都必须记录审计日志。

### 4. LLM Capability Boundaries

MVP LLM Provider 策略：

* MVP 不强制只能使用私有化或本地 LLM。
* 平台应兼容本地/私有化模型，以及 OpenAI 格式 API 和 Claude 格式 API 的外部 Provider。
* LLM Provider 必须由管理员显式配置、启用和停用，并记录配置变更审计。
* 不同 Provider 应共用同一套后端安全边界：不得接触明文弱密码，不得绕过后端权限、白名单和工具参数校验。
* 任意进入外部 Provider 的内容必须先经过脱敏策略处理，并记录 Provider、请求时间、调用目的、脱敏状态和任务 ID。
* 外部 Provider 可以处理意图识别、结构化任务计划、结果解释和报告草稿，但原始邮件样本、凭据、明文弱密码、未脱敏原始日志、Cookie、Token、URL 敏感参数等敏感内容不得直接发送给外部 Provider。

LLM 允许：

* 自然语言意图识别。
* 生成结构化任务计划。
* 解释扫描结果和工具失败原因。
* 合并重复漏洞。
* 生成风险摘要、处置建议和报告草稿。
* 分析钓鱼邮件语义、诱导话术和 IOC。
* 基于脱敏后的日志事件和聚合指标解释攻击趋势。
* 在需要澄清时生成明确问题。

LLM 不允许：

* 直接执行 `nmap`、`hydra`、`nuclei` 等扫描工具。
* 绕过资产白名单、权限、速率限制或超时限制。
* 自行扩大扫描范围。
* 接触明文弱密码。
* 读取未授权任务结果。
* 自动生成或替换爆破字典。
* 编造未由证据支持的漏洞结论。
* 将敏感原始样本、凭据、内网资产数据、未经脱敏的原始防火墙/Web 日志、Cookie、Token、URL 敏感参数或账号字段发送给未经批准的外部 LLM Provider。

### 5. Controlled Auxiliary Command Executor

* 平台可配置 LLM 允许执行的辅助命令白名单，例如 `ls`、`cd`、`pwd`、`base64`、`file`、`cat`。
* 辅助命令只能通过受控命令执行器运行，不能给 LLM 真实系统 shell。
* 每个任务必须使用独立工作目录，例如 `/workspace/tasks/{task_id}`。
* 辅助命令只能作用于当前任务工作目录。
* 禁止访问平台源码、系统目录、其他用户任务目录和凭据目录。
* 命令应按 argv 执行，不接受拼接完整 shell 字符串。
* 默认禁止管道、分号、逻辑操作符、命令替换、重定向和通配符展开。
* 每条命令必须有超时、输出大小限制和审计日志。
* `cd` 应实现为受控 working directory 切换，不是持久 shell 会话。
* 命令输出进入任务上下文前应脱敏。

### 6. Security Tool Executor

* 扫描类工具由后端任务编排器调用，不由 LLM 直接执行。
* MVP 工具集包括 `Subfinder`、`Httpx`、`Nmap`、`Nuclei`、`Hydra`、`emlAnalyzer`、`Magika`。
* 每个任务应隔离运行。
* 工具参数必须来自后端 schema，不接受任意 shell 字符串。
* 工具参数 schema 只允许后端预置字段、目标引用、强度档位和配置项引用；LLM 和用户都不得传入任意原生命令参数。
* 后端应根据 schema、工具强度和全局策略展开最终命令参数，并将展开后的参数写入审计日志。
* 管理员可配置工具路径和工具版本；每次执行必须记录工具名称、工具版本、配置来源和实际执行路径。
* 所有目标必须先通过资产组白名单校验。
* 所有输出应写入任务工作目录和结构化结果表。
* 工具执行应支持并发、速率、超时、资源限制和全局 kill switch。
* 工具应支持强度档位配置，尤其是 `Hydra`、`Nuclei`、`Nmap`。
* MVP 支持低、中、高三档工具强度。
* 低强度应作为默认档位，适用于常规资产发现、轻量存活探测、基础模板扫描和低并发验证。
* 中强度适用于授权资产组内的常规综合扫描，可提高并发、扩大端口范围或启用更完整模板集，但仍受全局速率和超时限制。
* 高强度适用于更深的端口扫描、更完整的 Nuclei 模板集或更高并发弱口令尝试，必须二次确认。
* 高强度不能被 YOLO 跳过确认，即使用户拥有 `task:yolo_execute` 权限。
* 高强度执行必须记录用户确认、强度档位、后端参数展开结果、速率限制、超时限制和资源限制。
* 后端必须为 `Hydra`、`Nuclei`、`Nmap` 分别维护强度档位到实际参数的映射；LLM 只能选择强度档位，不能直接写任意工具参数。
* 低/中/高档对应的具体并发、速率、超时、端口范围、模板集和爆破尝试上限作为配置项管理，不在 PRD 中固定数值。
* MVP 使用本机 Docker runner 执行扫描工具，避免主应用进程直接执行扫描工具。
* 扫描工具应在容器内运行，容器挂载当前任务工作目录并受资源、网络、超时和 kill switch 控制。
* 远程 runner 节点作为后续扩展，不作为 MVP 必需项。

### 7. Fixed Security Workflows

MVP 应以固定安全工作流为主。LLM 负责选择工作流和填参，不允许自由编排任意工具链。

#### 7.1 Phishing Email Analysis

* 支持邮件网关直连模式。
* 可选支持手动上传 `.eml` / `.msg` 或粘贴邮件原文。
* 邮件网关模式下，平台作为串联标记型邮件网关，邮件流向为外部邮件服务器 -> 平台网关 -> 企业真实邮箱服务器。
* MVP 邮件网关采用轻量串联转发模式，支持配置上游入站来源和下游真实邮箱服务器域名或 IP。
* 平台应自动接收入站邮件，完成验证和分析后再转发。
* 疑似钓鱼邮件应添加安全 Header 后转发，默认不删除、不隔离、不阻断。
* 建议安全 Header 包括 `X-Security-Phishing`、`X-Security-Risk-Score`、`X-Security-Task-ID` 和 `X-Security-Analysis`。
* MVP 钓鱼邮件风险评分采用 0-100 分：`>=80` 标记为 `suspected`，`50-79` 标记为 `suspicious` / medium，`<50` 标记为 `clean`。
* `X-Security-Risk-Score` 应写入 0-100 的整数风险分；`X-Security-Phishing` 应写入 `suspected`、`suspicious` 或 `clean`。
* 单封邮件超过大小上限时，不分析附件，只检测邮件正文，并添加邮件超限标签后继续转发。
* 邮件超限标签应通过安全 Header 标识，例如 `X-Security-Analysis: body-only-size-limit`。
* 分析服务不可用时应 fail-open，邮件继续转发并只写入 `X-Security-Analysis: unavailable`，不得输出钓鱼判断或风险分。
* MVP 不要求实现完整 MTA 级高可用、复杂邮件队列持久化或投递保障；备用 MX、转发失败重试和邮件队列持久化作为后续增强。
* 网关自身不可用时，应允许邮件系统走备用 MX 或备用转发路径。
* 分析能力应包括 SPF、DKIM、DMARC 验证，发件域信誉和 Header 异常检查，URL、附件、正文、IOC 提取，`Magika` 文件类型识别，`emlAnalyzer` 解析，以及 LLM 风险判断和处置建议。
* MVP 中附件和 URL 只做静态分析，不自动点击 URL，不自动打开附件，LLM 不作为 SMTP 投递决策的唯一判断来源。

#### 7.2 Asset Discovery

* 输入为授权根域名和授权资产组。
* 系统应使用 `Subfinder` 发现子域。
* 系统应使用 `Httpx` 探测存活、标题、状态码和技术栈。
* 结果应写入资产库。
* 超出白名单的发现结果应标记为待确认，不自动纳入可扫描范围。

#### 7.3 Port And Service Identification

* 输入为授权 IP、授权域名或授权资产组。
* 系统应使用 `Nmap` 执行端口和服务识别。
* 结果应结构化保存端口、协议、服务、版本和指纹。
* 结果应进入资产详情、漏洞扫描输入和大屏统计。

#### 7.4 Vulnerability Scanning

* 输入为授权资产、授权资产组和已发现服务。
* 系统应使用 `Nuclei` 按预配置模板集扫描。
* 系统应保存模板 ID、风险等级、命中证据和请求响应摘要。
* LLM 可合并重复漏洞并生成风险解释。
* 结果应进入漏洞列表、大屏和报告。

#### 7.5 Weak Password Scanning

* 输入为授权资产、服务类型和预配置字典。
* 安全工程师可在授权资产范围内直接发起弱口令扫描，不需要弱口令扫描专项发起权限；能发起该任务弱口令扫描即可在有效窗口内查看和导出该任务明文弱密码。
* 安全工程师可在任务执行中查看爆破出的明文弱密码。
* 明文弱密码不得落库。
* LLM 不得接触明文弱密码。
* 普通测试报告中的弱密码必须脱敏。
* 明文弱密码只允许在任务执行期间或任务完成后 30 分钟内查看和导出。
* 明文弱密码导出文件格式为 XLSX，文件必须使用每次导出随机生成的密码加密。
* 导出随机密码只在导出完成时向导出用户展示一次，不写入数据库、导出文件或审计日志。
* 过期后明文弱密码应彻底不可恢复，只保留脱敏结果。
* 查看、复制、导出明文弱密码必须记录审计日志，日志不能包含明文密码。

#### 7.6 Comprehensive Exposure Scan

* 示例指令：“检查资产组 A 的暴露风险。”
* 系统应按资产发现、存活探测、端口与服务识别、漏洞扫描、结果汇总、LLM 风险摘要的顺序执行。
* 如果某一步失败，应按任务状态模型处理，不得自动扩大扫描范围或提高攻击强度。

#### 7.7 Automatic Firewall And Web Log Analysis

* 平台应自动接入和分析防火墙日志、Web 日志，不依赖安全工程师手动上传或手动导入日志文件。
* MVP 日志源产品范围只承诺通用 Syslog、Nginx / Apache access log、Nginx / Apache error log 和 JSON log。
* 防火墙日志优先支持通用 Syslog 接收，包括 TCP、UDP、TLS Syslog。
* 具体防火墙 / WAF 厂商格式、CEF、LEEF 和厂商 key-value 格式作为可扩展解析器，不作为 MVP 必须兼容项。
* Web 日志优先支持 Nginx、Apache 的 access log、error log、JSON log 或 Syslog 转发；反向代理和 WAF 日志通过通用 JSON log 或后续扩展解析器接入。
* MVP 采用平台提供 Syslog / 日志接收端，并由防火墙、WAF、Nginx / Apache 主动转发日志的方式。
* 轻量采集 Agent 作为可选扩展点保留，不作为 MVP 必需项。
* 系统应配置日志源、日志类型、解析格式、所属资产组、脱敏结构化事件保留周期和聚合指标保留周期。
* 日志接收服务应在接收时完成解析、脱敏和结构化，不保留原始日志正文，并生成可追溯事件 ID。
* 原始日志只允许在接收解析链路中短暂处理；脱敏结构化事件落库后应丢弃原始日志正文。
* 日志解析器应归一化时间、来源 IP、目标 IP / 域名、协议、端口、动作、规则 ID、严重级别、URL、HTTP 方法、状态码、User-Agent 等字段。
* 攻击识别规则应分类扫描探测、暴力破解、SQL 注入、XSS、目录遍历、命令执行、敏感路径访问、异常错误激增、WAF / 防火墙规则命中等事件。
* 聚合服务应按时间窗口、攻击类型、来源、目标资产、动作、严重级别生成趋势指标。
* LLM 只能基于脱敏后的结构化事件或聚合结果生成解释和摘要。
* 日志分析不得等同于 SIEM 深度关联分析，不自动攻击溯源，不生成攻击者画像，不因日志中出现新目标而自动扩大资产白名单或扫描范围，不基于日志自动执行封禁、阻断、扫描或漏洞利用。

### 8. Log Analysis Data Contract

* 日志源配置至少包含 `source_id`、日志类型、防火墙 / WAF / Web 服务产品类型、接入协议、解析格式、所属资产组、启停状态、脱敏结构化事件保留周期和聚合指标保留周期。
* Syslog 接入应区分 TCP、UDP、TLS，并记录监听地址、端口、证书配置和来源 IP 允许列表。
* Web 日志接入应记录 Nginx / Apache / 反向代理 / WAF 的日志格式版本，避免解析器隐式猜测字段含义。
* 日志源创建、修改、停用、删除和解析格式变更必须进入审计日志。
* 原始日志正文不落库、不进入对象存储、不支持查看、检索或导出。
* 日志接收元数据至少包含 `ingest_ref`、`source_id`、接收时间、原始事件时间、原始大小、校验摘要、截断标记、解析状态、脱敏状态和原文丢弃状态。
* 脱敏结构化事件默认保留 180 天，聚合指标默认保留 1 年；管理员可按日志源覆盖配置。
* 日志事件导出只允许导出脱敏后的结构化事件和聚合指标，并必须校验权限、记录审计。
* 归一化日志事件至少包含 `event_id`、`ingest_ref`、`source_id`、日志类型、事件时间、接收时间、来源 IP / 端口、目标 IP / 域名 / 端口、协议、动作、规则 ID / 规则名、严重级别。
* Web 事件应额外包含 HTTP 方法、URI 路径、状态码、User-Agent 摘要、请求大小和响应大小；Cookie、Token、账号、完整查询参数等敏感字段不得默认展示。
* 攻击分类字段应包含攻击类型、分类规则 ID、置信度和说明。
* 目标资产字段应包含 `target_asset_id` 或未命中状态。
* 趋势聚合结果至少包含时间窗口、日志类型、攻击类型、严重级别、来源 IP / 网段、目标资产、处置动作和事件数量。
* 大屏只消费聚合结果和脱敏后的结构化事件，不读取原始日志正文。
* 聚合统计应支持空窗口展示；采集失败应通过日志源健康状态单独展示。

MVP 数据存储边界：

* 结构化任务结果、资产、漏洞、弱口令脱敏结果、脱敏日志事件、聚合指标、任务状态和审计日志应进入关系型数据库。
* 任务原始证据、工具 stdout / stderr、附件静态分析产物和报告文件应进入对象存储或本地文件存储，并在数据库中保存引用。
* 日志原文不存储；日志链路只保留接收元数据、脱敏结构化事件和聚合指标。

### 9. Task Status Model

任务状态至少包括：

* `SUCCESS`：所有计划步骤完成，结果已结构化入库。
* `PARTIAL_SUCCESS`：部分工具成功、部分失败，成功结果仍展示，失败步骤明确标注。
* `FAILED`：关键前置步骤失败，导致后续无法执行。
* `NEEDS_CLARIFICATION`：LLM 识别不确定或缺少关键字段。
* `BLOCKED`：权限、白名单或安全策略阻断。
* `CANCELLED`：用户或 kill switch 主动终止。

失败路径要求：

* 工具失败时保留 stdout、stderr、退出码、工具版本、执行参数。
* 网络超时或临时失败可做有限重试，例如 1-2 次。
* 部分成功任务应产出可用结果，但报告状态必须为部分成功。
* 部分成功结果要明确标注失败步骤和数据不完整。
* Kill switch 只停止扫描工具、辅助命令和任务编排中的运行中步骤。
* Kill switch 触发后，已完成步骤的结构化结果和证据应保留，任务状态置为 `CANCELLED`。
* 被 kill switch 取消的原任务不允许恢复执行；用户需要重新创建任务。
* Kill switch 不影响邮件网关转发链路和日志接收链路。
* 不因失败自动扩大扫描范围。
* 不因失败自动提高攻击强度。
* 不因失败自动切换或扩大爆破字典。

### 10. Reports And Exports

* 支持任务报告导出。
* 支持资产风险报告导出。
* 支持钓鱼邮件分析报告导出。
* 支持漏洞扫描报告导出。
* 支持弱口令扫描报告导出。
* 支持日志攻击态势报告导出，只导出脱敏结构化事件和聚合指标；原始日志不保留也不可导出。
* MVP 报告导出格式只支持 PDF 和 HTML。
* MVP 报告模板采用内置固定模板，不提供自定义模板编辑能力。
* 报告语言默认中文。
* 所有报告默认脱敏，不包含明文弱密码、原始日志正文、Cookie、Token、账号、URL 敏感参数等敏感原文。
* MVP 不做报告签章、水印或防篡改校验。
* 普通弱口令测试报告中弱密码必须脱敏。
* 明文弱密码不落库。
* 明文弱密码导出只能在执行期或任务完成后 30 分钟内完成。
* 明文弱密码导出格式为随机密码加密的 XLSX 文件。
* 明文导出必须记录审计日志，但审计日志不能包含明文密码。

### 11. Dashboard

MVP 大屏应展示 7 类指标：

* 任务态势：今日任务数、运行中任务数、成功/部分成功/失败/阻断任务数、平均耗时。
* 资产态势：授权资产组数量、发现资产数、存活资产数、新增资产、暴露端口、Top 服务。
* 漏洞态势：高危/中危/低危漏洞数量、Top 漏洞类型、Top 风险资产、Nuclei 模板命中趋势。
* 弱口令态势：弱口令资产数、服务类型分布、趋势，不展示明文密码。
* 钓鱼邮件态势：今日邮件量、疑似钓鱼量、风险评分分布、Top 诱导类型、Top URL / 域名、附件类型分布。
* YOLO / 智能体态势：自然语言任务数、YOLO 直接执行数、澄清次数、被白名单阻断次数。
* 日志攻击态势：防火墙日志事件量、Web 日志事件量、攻击趋势图、Top 攻击类型、Top 来源 IP / 网段、Top 目标资产、防火墙 / WAF 动作分布、Web 攻击 URI / HTTP 方法 / 状态码分布、新增异常峰值和持续攻击趋势提示。
* 大屏按准实时方式刷新，默认刷新间隔为 60 秒。
* 日志异常峰值提示按最近 15 分钟窗口与前 1 小时基线对比，事件量超过基线 3 倍时提示。

明确不做：

* 3D 地球攻击动画。
* 无业务价值的动态粒子效果。
* 复杂 SIEM 关联分析。
* 自动攻击者画像。
* 基于日志的自动封禁、自动反制或自动扩大扫描范围。

### 12. Audit

所有关键动作必须可追溯，包括：

* 用户原始自然语言指令。
* LLM 意图识别结果。
* LLM 生成的结构化任务计划。
* 用户是否开启 YOLO。
* 澄清和确认记录。
* 权限校验结果。
* 资产白名单命中结果。
* 工具名称、版本、参数、开始时间、结束时间。
* 任务 stdout、stderr、退出码。
* 任务状态变更。
* 报告导出记录。
* 明文弱密码查看、复制、导出记录。
* 日志源创建、修改、停用和删除记录。
* 日志解析规则、攻击分类规则和保留周期变更记录。

## Acceptance Criteria

### Scenario 1: Natural Language Task Creation

给定安全工程师已登录并拥有资产组 A 权限，当输入“扫描资产组 A 的暴露风险”时：

* 系统应识别为综合扫描任务。
* 系统应生成结构化任务计划。
* 系统应校验资产组白名单。
* YOLO 开启时应直接创建并执行任务。
* 任务结果应能在任务详情、大屏和报告中查看。

### Scenario 2: Authorization And Whitelist Block

给定用户没有资产组 B 权限，当输入“扫描资产组 B”时：

* 系统应拒绝执行。
* 任务状态应为 `BLOCKED`。
* 系统应记录阻断原因和审计日志。
* 不应调用任何扫描工具。

### Scenario 3: YOLO Clarification

给定 YOLO 已开启，当用户输入“看看公司外网有没有问题”且系统无法确定具体资产组时：

* 系统不得猜测执行。
* 系统应在对话栏展示待确认操作或要求选择资产组。
* 用户一次点击确认后才能继续。

### Scenario 4: Phishing Mail Gateway

给定平台作为串联标记型邮件网关接收入站邮件时：

* 系统应执行 SPF、DKIM、DMARC 和邮件内容分析。
* 系统应调用 `emlAnalyzer` 和 `Magika`。
* 疑似钓鱼邮件应添加安全 Header 后转发。
* 非钓鱼邮件应正常转发。
* 分析服务不可用时应 fail-open，邮件继续转发并添加分析失败标签。

### Scenario 5: Comprehensive Scan Toolchain

给定用户对授权资产组发起综合扫描时：

* 系统应至少完成 `Subfinder -> Httpx -> Nmap -> Nuclei` 链路。
* 每一步结果应结构化入库。
* 原始证据应可追溯。
* 工具参数应来自受控 schema。
* LLM 不得直接执行扫描工具命令。

### Scenario 6: Weak Password Scan

给定安全工程师对授权资产发起弱口令扫描时：

* 系统应调用 `Hydra` 执行扫描。
* 明文弱密码可在任务执行期或短时间窗口内查看。
* 明文弱密码不得落库。
* LLM 不得接触明文弱密码。
* 普通报告中弱密码必须脱敏。
* 明文弱密码导出不需要独立导出权限；能发起该任务弱口令扫描即可在有效窗口内导出，并必须记录审计。

### Scenario 7: Partial Success

给定综合扫描中 `Subfinder` 和 `Httpx` 成功但 `Nmap` 超时：

* 任务状态应为 `PARTIAL_SUCCESS`。
* 成功结果应展示。
* 失败步骤应标红或明确标注。
* 报告状态应显示部分成功。
* 系统应保留失败工具的 stderr、退出码、执行参数和时间。

### Scenario 8: LLM Boundaries

给定 LLM 生成了扫描计划：

* 后端必须重新校验权限、白名单和参数 schema。
* LLM 只能通过结构化任务间接触发扫描工具。
* LLM 只能执行配置白名单内的辅助命令。
* 辅助命令只能在当前任务隔离目录运行。
* 不允许 LLM 拼接任意 shell 字符串执行。

### Scenario 9: Real Data Dashboard

给定系统已有真实任务、资产、漏洞、弱口令、邮件分析和日志分析数据：

* 大屏应展示任务、资产、漏洞、弱口令、钓鱼邮件、智能体执行和日志攻击态势 7 类指标。
* 指标应来源于真实任务数据。
* 弱口令大屏不得展示明文密码。
* 日志攻击态势应来源于自动接入的防火墙日志和 Web 日志，不依赖手动导入。

### Scenario 10: Audit Closure

给定任意任务创建、执行、失败、导出、阻断或 kill switch 操作：

* 系统应能追溯操作者、时间、输入、计划、权限校验、工具调用、结果和状态变化。
* 明文弱密码相关审计不得包含明文密码。

### Scenario 11: Automatic Log Analysis And Attack Trend Dashboard

给定平台已配置防火墙日志源和 Web 日志源，且日志源持续发送日志：

* 系统应自动接收日志，不要求安全工程师手动上传或导入日志文件。
* 系统不保留原始日志正文，应保存接收元数据、脱敏结构化事件和聚合指标。
* 系统应识别常见攻击类型和防火墙 / WAF 处置动作。
* 系统应按时间窗口聚合攻击趋势，并在大屏展示趋势图。
* 大屏应展示 Top 攻击类型、Top 来源、Top 目标资产、动作分布和 Web 攻击 URI / 状态码分布。
* 未命中资产白名单或资产库的日志目标不得自动纳入可扫描范围。
* LLM 生成日志摘要时不得接触未经脱敏的敏感原始日志。

## Definition Of Done For PRD Phase

* PRD 覆盖任务说明中的用户、范围、工作流、安全边界、验收场景和模块拆解。
* 所有不明确内容单独列入 `Ambiguities To Resolve Before Implementation`。
* 不启动实现阶段，不运行 `task.py start`。
* 用户确认 PRD 方向后，再进入后续需求澄清或 Trellis 拆解。
